# DEC-001: Adopt a Consolidated Maestro Workspace Layout

**Status:** Accepted
**Date:** 2026-04-24
**Decider:** dnvt with Codex facilitation

## Question

Should Maestro adopt a single `.maestro` workspace with `source`, `project`,
`state`, and `generated` subdirectories, plus a layout reconciliation script?

## First-Principles Frame

We are not deciding whether the repo root should look cleaner. We are deciding
where Maestro's source, project contract, runtime state, and generated adapter
surfaces should live so AI workflows remain reliable without spreading
workflow-only files across every consumer repository.

The question does constrain the options by naming a preferred shape. To avoid
rubber-stamping it, the viable alternatives below include status quo, a lighter
root-cleanup variant, and an external global-state model.

If this decision is wrong, the assumption that breaks is: adapter tooling can
reliably load generated projections from a consolidated internal workspace. The
observation that would surface the breakage is direct: Claude or Codex commands
stop loading, agent/rule discovery fails, generated files drift from source, or
new contributors cannot distinguish source, state, generated output, and safe
cache.

## Door Type

This is a two-way door during compatibility work and a costly-to-reverse choice
once new installs default to the layout. Treat Phase 1 as reversible and Phase 2
as release-gated.

## Options Considered

### Option A: Keep the Current Root-Spread Layout

Keep `.maestro`, `.maestro-project`, `maestro.toml`, `.claude`, `progress`,
`plans`, `analysis`, `docs/decisions`, `AGENTS.md`, `CLAUDE.md`, and
`MEMORY.md` at the repo root, with `.gitignore` hiding generated/runtime files.

This is simple and compatible with current generated paths, but it keeps root
noise high and makes it hard to know which files are source, state, generated,
or user-owned.

### Option B: Single `.maestro` Workspace With Projections

Adopt one workspace:

```text
.maestro/
  source/
  project/
  state/
  generated/
  cache/
  backups/
```

Keep only required adapter projections at the root, such as `.claude/` or
`AGENTS.md`, and treat those projections as disposable generated output.

This creates the clearest mental model and gives Maestro one durable home, but
requires migration logic and compatibility handling.

### Option C: Keep `.maestro` as Source, Move Only Runtime State

Leave the Maestro submodule at `.maestro`, move `progress`, `plans`,
`analysis`, and decisions under an ignored state directory, and keep existing
root config and generated adapters.

This reduces some noise with less migration cost, but it keeps the source/state
boundary ambiguous because `.maestro` itself remains both product source and
workflow home.

### Option D: Store Runtime State Outside the Repo

Keep Maestro source and generated adapters in the repo, but move state to a
global location such as `~/.maestro/projects/<repo-id>`.

This makes repo roots clean, but hurts portability, repo-local reviewability,
and task handoff. It also creates identity and sync problems when a repo is
cloned to multiple paths or machines.

## Criteria

- **Root hygiene:** workflow internals should not dominate consumer repos.
- **Tool reliability:** Claude, Codex, and future adapters must keep loading.
- **Source/state separation:** shared Maestro source must not contain consumer
  runtime state.
- **Migration safety:** existing repos need dry-run plans, backups, and no
  deletion of unknown tracked files.
- **Portability:** a repo clone should contain enough local context to restore
  workflow state when desired.
- **Extensibility:** the layout should support more adapters without adding more
  root-level directories.

## Decision

Adopt Option B: a consolidated `.maestro` workspace with `source`, `project`,
`state`, and `generated` subdirectories, plus an idempotent layout
reconciliation script.

New installs should target this layout:

```text
.maestro/
  source/                 # pinned Maestro source, usually a submodule
  project/                # tracked repo-local contract and overrides
    maestro.toml
    overrides/
  state/                  # ignored local/runtime workflow state
    context/
    progress/
    plans/
    analysis/
    decisions/
    memory/
  generated/              # ignored generated adapter output
    claude/
    codex/
    reports/
  cache/                  # ignored derived cache
  backups/                # ignored migration backups
```

Root projections are allowed only when external tools require them. Examples:

```text
.claude/                  # generated or mirrored adapter surface
AGENTS.md                 # optional generated Codex projection
```

The reconciliation script should provide:

```bash
.maestro/source/scripts/reconcile-layout.sh check
.maestro/source/scripts/reconcile-layout.sh plan
.maestro/source/scripts/reconcile-layout.sh migrate --apply
.maestro/source/scripts/reconcile-layout.sh clean --apply
```

It should detect legacy paths, classify them as tracked, ignored, generated, or
unknown, produce a plan before moving anything, back up moved files under
`.maestro/backups/`, regenerate adapter projections, and remove stale generated
files only when they contain a Maestro generated marker.

## Consequences

Maestro needs a phased migration.

Phase 1 keeps compatibility. Existing `.maestro/generate.sh` and commands
should support both old and new layouts, read the new layout config when
present, and warn when legacy root paths exist.

Phase 2 flips the default. New installs should use `.maestro/source`,
`.maestro/project`, `.maestro/state`, and `.maestro/generated`. Existing repos
should migrate through `maestro:health --fix-layout` or
`maestro:doctor --fix-layout`, backed by `reconcile-layout.sh`.

Generated files must carry a source/checksum marker. Every generated adapter
surface must be disposable. The shared source directory must never store
consumer runtime state.

## Evidence Quality

Evidence is medium strength. The root-noise problem is directly observed in
this repo and adjacent Maestro-enabled projects. The adapter-reliability claim
is still an assumption until the generator proves that `.claude`, Codex, and
future projections remain stable after migration.

## Revisit Trigger

Reopen this decision if any of these occur:

- Claude or Codex cannot load generated commands, agents, rules, or skills from
  the projected layout.
- The migration script creates dirty submodule state or loses workflow context.
- Contributors cannot explain which files are source, project config, runtime
  state, generated output, or cache after using the new layout.
- A new required adapter forces multiple persistent root directories that cannot
  be modeled as generated projections.

## Next Step

Create an implementation plan for Maestro layout v2 compatibility and the
`reconcile-layout.sh` script before changing the default install layout.
